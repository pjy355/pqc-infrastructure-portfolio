# OpenSSL 3.5 / 3.6 / 4.0 비교 분석

세 버전을 ARM64 Ubuntu 22.04 환경에서 직접 빌드하고 실제로 사용해보면서 어떤 차이가 있는지 확인했다. 목적은 회사 PQC 인프라에 어떤 버전을 기준으로 삼을지 결정하는 것이었다.

---

## 왜 비교가 필요했나

PQC를 적용하려면 ML-KEM을 지원하는 OpenSSL이 필요했다. liboqs와 oqs-provider를 붙여서 쓰는 방법도 있었지만, OpenSSL 자체에 ML-KEM이 내장된 버전이 있다면 그쪽이 훨씬 의존성이 단순해진다. 그래서 3.5, 3.6, 4.0을 각각 받아서 빌드부터 다시 해봤다.

---

## OpenSSL 3.5

가장 먼저 빌드했고, 결과적으로 최종 선택도 이 버전이 됐다.

ML-KEM-512, ML-KEM-768, ML-KEM-1024가 모두 기본으로 들어있어서 `openssl list -kem-algorithms`로 바로 확인된다. liboqs나 oqs-provider를 따로 설치할 필요가 없다는 뜻이고, 이게 빌드 시간과 의존성 관리 양쪽에서 꽤 큰 차이를 만든다. 외부 provider를 붙이면 버전 호환성 문제나 빌드 실패 가능성이 늘어나는데, 3.5는 그 단계를 건너뛸 수 있었다.

빌드 자체도 가장 매끄러웠다. `./config` 단계에서 별다른 옵션 충돌 없이 진행됐고, 이후 strongSwan과 연동할 때도(RPATH, PKG_CONFIG_PATH 설정만 맞춰주면) 문제없이 붙었다.

---

## OpenSSL 3.6

3.5와 큰 틀에서는 비슷하지만, LMS 알고리즘 쪽에서 제약이 있었다. LMS가 검증(verification) 전용으로만 동작하고, 인증서를 생성하거나 서명하는 작업에는 쓸 수 없었다. 실제로 서명 관련 작업을 시도하면 `unrecognized algorithm` 오류가 나는 경우가 있었다.

PQC 인프라에서는 인증서 발급(서명)이 필수적인 작업이라, 이 제약 하나만으로도 3.6을 메인으로 쓰기는 어렵다고 판단했다.

---

## OpenSSL 4.0

가장 변화가 큰 버전이었다. SSLv3 같은 레거시 API가 완전히 제거되어서, 옛날 코드를 그대로 가져다 쓰면 마이그레이션이 필요하다. ENGINE API도 제거되어 커스텀 엔진을 붙이는 방식의 통합은 더 이상 불가능하다.

대신 ECH(Encrypted Client Hello)가 추가된 점은 눈에 띄었다. TLS 핸드셰이크에서 ClientHello 자체를 암호화하는 기능인데, 앞으로 TLS 스택이 어느 방향으로 가는지 보여주는 신호 같았다. 다만 지금 당장의 PQC + strongSwan 파이프라인 구축에는 레거시 API 제거로 인한 마이그레이션 부담이 더 크게 느껴졌고, 아직 프로덕션에 바로 쓰기엔 시기상조라고 봤다.

---

## 비교 요약

| 항목 | OpenSSL 3.5 | OpenSSL 3.6 | OpenSSL 4.0 |
|------|------------|------------|------------|
| ML-KEM 내장 | 있음 (512/768/1024) | 있음 | 있음 |
| liboqs/oqs-provider 필요 여부 | 불필요 | 불필요 | 불필요 |
| LMS 알고리즘 | 생성/검증 모두 가능 | 검증 전용 (서명 불가) | 생성/검증 모두 가능 |
| SSLv3 / 레거시 API | 유지 | 유지 | 완전 제거 |
| ENGINE API | 유지 | 유지 | 제거 |
| ECH 지원 | 없음 | 없음 | 있음 |
| 빌드 안정성 | 가장 안정적 | 안정적 | 일부 마이그레이션 필요 |

---

## 결론

OpenSSL 3.5를 기준 버전으로 결정했다. ML-KEM을 외부 의존성 없이 완전하게 지원하면서도, 인증서 서명 같은 PKI 핵심 작업에 제약이 없고, 빌드와 strongSwan 연동 과정에서 가장 적은 마찰을 보였다. 3.6은 LMS 제약 때문에, 4.0은 레거시 API 제거로 인한 마이그레이션 부담 때문에 현재 시점에서는 후순위로 두었다.

이 결정은 이후 회사 서버에 PQC + strongSwan 파이프라인을 구축할 때 그대로 적용했다.
